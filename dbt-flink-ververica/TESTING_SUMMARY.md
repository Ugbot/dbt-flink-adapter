### dbt-flink-ververica CLI

#### `compile` Command
```bash
dbt-flink-ververica compile \
  --project-dir test-vvc-deployment \
  --profiles-dir test-vvc-deployment \
  --target dev
```

**What it does**:
1. Runs `dbt compile` to generate compiled SQL
2. Reads models from `target/compiled/`
3. Strips dbt-flink query hints (`/** mode('batch') */`)
4. Generates SET statements from hints
5. Extracts DROP statements
6. Writes clean SQL to `target/ververica/`

**Output**:
```

✓ Found 3 compiled models
✓ Processed 3 models

Writing processed SQL to: test-vvc-deployment/target/ververica

  ✓ simple_test.sql
  ✓ test_pipeline.sql
  ✓ dbt_flink-on-run-start-0.sql

✓ Compilation complete!
```

## Bugs Fixed 🐛

### 1. AdapterResponse Type Error

**Error**:
```python
AttributeError: 'str' object has no attribute 'query_id'
```

**Root Cause**: The `get_response()` method in `dbt/adapters/flink/connections.py` was returning a plain string instead of an `AdapterResponse` object.

**Fix** (`dbt/adapters/flink/connections.py:167-184`):
```python
from dbt.adapters.contracts.connection import AdapterResponse

@classmethod
def get_response(cls, cursor: FlinkCursor) -> AdapterResponse:
    status = cursor.get_status()
    
    # Get operation handle as query_id if available
    query_id = None
    if cursor.last_operation is not None:
        query_id = cursor.last_operation.operation_handle
    
    return AdapterResponse(
        _message=status,
        code=status,
        query_id=query_id,
    )
```

**Impact**: dbt-core 1.8+ requires proper `AdapterResponse` objects for query tracking and logging.

### 2. Wrong Artifact Directory

**Error**:
```
dbt run directory not found: test-vvc-deployment/target/run
Run 'dbt compile' first to generate compiled artifacts.
```

**Root Cause**: `DbtArtifactReader` was looking for `target/run/` but dbt creates compiled artifacts in `target/compiled/`.

**Fix** (`src/dbt_flink_ververica/sql_processor.py:430`):
```python
class DbtArtifactReader:
    def __init__(self, project_dir: Path, target: str = "dev"):
        self.project_dir = project_dir
        self.target = target
        # Changed from target/run to target/compiled
        self.compiled_dir = project_dir / "target" / "compiled"
    
    def find_compiled_models(self, models: Optional[List[str]] = None):
        if not self.compiled_dir.exists():
            raise FileNotFoundError(
                f"dbt compiled directory not found: {self.compiled_dir}\n"
                f"Run 'dbt compile' first to generate compiled artifacts."
            )
        
        # Find all .sql files in target/compiled
        for sql_file in self.compiled_dir.rglob("*.sql"):
            # ...
```

**Impact**: The CLI now correctly reads from dbt's compiled artifacts directory.

## Known Issues ⚠️

### Streaming Table Materialization Bug

**Status**: Identified in dbt-flink-adapter, not in dbt-flink-ververica

**Description**: The `streaming_table` materialization with explicit `schema` config parameter fails with SQL parse errors.

**Example**:
```sql
{{ config(
    materialized='streaming_table',
    execution_mode='streaming',
    schema='''
        user_id BIGINT,
        event_type STRING,
        amount DECIMAL(10, 2),
        event_time TIMESTAMP(3)
    ''',
    watermark={'column': 'event_time', 'strategy': 'event_time - INTERVAL \'5\' SECOND'},
    properties={'connector': 'datagen', 'rows-per-second': '10'}
) }}

SELECT * FROM (VALUES (1, 'test', 100.00, CURRENT_TIMESTAMP)) AS t(...)
WHERE FALSE
```

**Error**:
```
SQL parse failed. Encountered "BIGINT" at line 1, column 40.
Was expecting one of: <EOF> "WITH" ... "COMMENT" ... ";" ... "." ...
```

**Root Cause**: The adapter is misinterpreting the `schema` parameter as a database schema name instead of table column definitions. It tries to execute:
```sql
CREATE DATABASE IF NOT EXISTS _user_id BIGINT, event_type STRING...
```

**Workaround**: Use simple `CREATE TABLE AS` patterns without explicit schema definitions for now.

**Working Pattern**:
```sql
{{ config(materialized='table') }}

SELECT 
    id,
    name,
    amount,
    created_at
FROM (VALUES (1, 'test', 100.00, CURRENT_TIMESTAMP)) AS t(id, name, amount, created_at)
```

**Needs Investigation**: This is a bug in the dbt-flink adapter's `streaming_table` materialization macro, not in dbt-flink-ververica.

## Lessons Learned

### 1. dbt-core 1.8+ Breaking Changes

The adapter decoupling in dbt-core 1.8+ requires strict type adherence:
- Must return `AdapterResponse` objects, not strings
- Must include `query_id` field for query tracking
- Logging and telemetry depend on proper response types

### 2. dbt Artifact Structure

`dbt compile` creates artifacts in `target/compiled/`, not `target/run/`:
- `target/compiled/` contains Jinja-rendered SQL (no query results)
- `target/run/` contains executed queries with results (only after `dbt run`)
- For SQL generation, always read from `target/compiled/`

### 3. Streaming Table Limitations

The dbt-flink adapter's `streaming_table` materialization needs work:
- Explicit `schema` config parameter conflicts with internal logic
- Watermark definitions work in isolation but not with schema
- Connector properties (DataGen, Kafka) need special handling
- Batch `CREATE TABLE AS` patterns are more reliable for now

### 4. Flink SQL Gateway Stability

Flink 1.20 SQL Gateway is production-ready:
- Session management works well with heartbeats
- REST API is reliable and fast
- Error messages are detailed and helpful
- Good compatibility with dbt's query execution model

### 5. CLI Design Patterns

**What Worked Well**:
- Typer for CLI framework (automatic help, validation)
- Pydantic for configuration (type safety, validation)
- Rich for progress indicators (spinners, colors, tables)
- HTTPX for async API calls (clean, modern)
- Keyring for credential storage (secure, cross-platform)

**What Needs Improvement**:
- Better error messages when dbt compile fails
- Progress indicators for long-running operations
- Retry logic for API calls
- Validation of SQL before deployment
